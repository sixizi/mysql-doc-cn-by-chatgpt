#### 8.4.1.5 PAM 可插拔身份验证

> **注意**
>
> PAM 可插拔认证是 MySQL 企业版中的一个扩展功能，这是一款商业产品。要了解更多关于商业产品的信息，请参见 https://www.mysql.com/products/。

MySQL 企业版支持一种认证方法，使 MySQL 服务器可以使用 PAM（可插拔认证模块）来认证 MySQL 用户。PAM 使系统可以使用标准接口访问各种认证方法，如传统的 Unix 密码或 LDAP 目录。

PAM 可插拔认证提供以下功能：

- **外部认证**：PAM 认证使 MySQL 服务器可以接受从 MySQL 授权表之外定义的用户连接，并使用 PAM 支持的方法进行认证。
- **代理用户支持**：PAM 认证可以根据外部用户所属的 PAM 组和提供的认证字符串，向 MySQL 返回一个与客户端程序传递的外部用户名不同的用户名。这意味着插件可以返回定义了外部 PAM 认证用户应具有的权限的 MySQL 用户。例如，一个名为 joe 的操作系统用户可以连接并拥有名为 developer 的 MySQL 用户的权限。

PAM 可插拔认证已在 Linux 和 macOS 上进行了测试；请注意，Windows 不支持 PAM。

下表显示了插件和库文件的名称。文件名后缀可能因系统而异。文件必须位于 `plugin_dir` 系统变量指定的目录中。有关安装信息，请参见“安装 PAM 可插拔认证”。

表 8.20 PAM 认证的插件和库名称

| 插件或文件   | 插件或文件名            |
| ------------ | ----------------------- |
| 服务器端插件 | `authentication_pam`    |
| 客户端插件   | `mysql_clear_password`  |
| 库文件       | `authentication_pam.so` |

与服务器端 PAM 插件通信的客户端明文插件 `mysql_clear_password` 内置于 `libmysqlclient` 客户端库中，并包含在所有发行版中，包括社区版。所有 MySQL 发行版中都包含客户端明文插件，使任何发行版的客户端都可以连接到加载了服务器端 PAM 插件的服务器。

##### PAM 认证 MySQL 用户的工作原理

本节概述了 MySQL 和 PAM 如何协作认证 MySQL 用户。有关如何设置 MySQL 账户以使用特定 PAM 服务的示例，请参见“使用 PAM 可插拔认证”。

客户端程序和服务器进行通信，客户端将客户端用户名（默认情况下为操作系统用户名）和密码发送到服务器：

- 客户端用户名是外部用户名。
- 对于使用 PAM 服务器端认证插件的账户，相应的客户端插件是 `mysql_clear_password`。此客户端插件不进行密码哈希，因此客户端以明文形式将密码发送到服务器。

服务器根据外部用户名和客户端连接的主机查找匹配的 MySQL 账户。PAM 插件使用 MySQL 服务器传递给它的信息（如用户名、主机名、密码和认证字符串）。当您定义使用 PAM 认证的 MySQL 账户时，认证字符串包含：

- PAM 服务名，即系统管理员用于引用特定应用程序的认证方法的名称。单个数据库服务器实例可以关联多个应用程序，因此服务名的选择留给 SQL 应用程序开发人员决定。
- 可选地，如果要使用代理，则需要一个从 PAM 组到 MySQL 用户名的映射。

插件使用认证字符串中指定的 PAM 服务名检查用户凭证，并返回“认证成功，用户名是 `user_name`”或“认证失败”。密码必须适用于 PAM 服务使用的密码存储。例如：

- 对于传统的 Unix 密码，服务查找存储在 `/etc/shadow` 文件中的密码。
- 对于 LDAP，服务查找存储在 LDAP 目录中的密码。

如果凭证检查失败，服务器将拒绝连接。

否则，认证字符串指示是否发生代理。如果字符串不包含 PAM 组映射，则不会发生代理。在这种情况下，MySQL 用户名与外部用户名相同。

否则，基于 PAM 组映射指示代理，MySQL 用户名基于映射列表中的第一个匹配组确定。“PAM 组”的含义取决于 PAM 服务。例如：

- 对于传统的 Unix 密码，组是定义在 `/etc/group` 文件中的 Unix 组，可能会补充 `/etc/security/group.conf` 文件中的其他 PAM 信息。
- 对于 LDAP，组是定义在 LDAP 目录中的 LDAP 组。

如果代理用户（外部用户）对代理的 MySQL 用户名具有 PROXY 权限，则会发生代理，代理用户将获得代理用户的权限。

##### 安装 PAM 可插拔认证

本节描述如何安装服务器端 PAM 认证插件。有关安装插件的一般信息，请参见 [7.6.1 节, “安装和卸载插件”](#7.6.1-安装和卸载插件)。

要使服务器能够使用插件库文件，文件必须位于 MySQL 插件目录中（由 `plugin_dir` 系统变量指定的目录）。如有必要，通过在服务器启动时设置 `plugin_dir` 的值来配置插件目录位置。

插件库文件的基本名称是 `authentication_pam`，通常编译后带有 `.so` 后缀。

要在服务器启动时加载插件，请使用 `--plugin-load-add` 选项指定包含它的库文件。使用这种插件加载方法，每次服务器启动时都必须给出该选项。例如，将这些行放入服务器的 `my.cnf` 文件中：

```ini
[mysqld]
plugin-load-add=authentication_pam.so
```

修改 `my.cnf` 后，重新启动服务器以使新设置生效。

或者，要在运行时加载插件，请使用此语句，必要时调整 `.so` 后缀：

```sql
INSTALL PLUGIN authentication_pam SONAME 'authentication_pam.so';
```

`INSTALL PLUGIN` 立即加载插件，并在 `mysql.plugins` 系统表中注册它，使服务器在每次后续正常启动时加载它，而无需 `--plugin-load-add`。

要验证插件安装，请检查 Information Schema PLUGINS 表或使用 `SHOW PLUGINS` 语句（参见 [7.6.2 节, “获取服务器插件信息”](#7.6.2-获取服务器插件信息)）。例如：

```sql
mysql> SELECT PLUGIN_NAME, PLUGIN_STATUS
       FROM INFORMATION_SCHEMA.PLUGINS
       WHERE PLUGIN_NAME LIKE '%pam%';
+--------------------+---------------+
| PLUGIN_NAME        | PLUGIN_STATUS |
+--------------------+---------------+
| authentication_pam | ACTIVE        |
+--------------------+---------------+
```

如果插件未能初始化，请检查服务器错误日志中的诊断消息。

要将 MySQL 账户与 PAM 插件关联，请参见“使用 PAM 可插拔认证”。

##### 卸载 PAM 可插拔认证

卸载 PAM 认证插件的方法取决于您如何安装它：

- 如果您在服务器启动时使用 `--plugin-load-add` 选项安装插件，请在没有该选项的情况下重新启动服务器。
- 如果您在运行时使用 `INSTALL PLUGIN` 语句安装插件，则插件在服务器重启后仍保持安装状态。要卸载它，请使用 `UNINSTALL PLUGIN`：

```sql
UNINSTALL PLUGIN authentication_pam;
```

##### 使用 PAM 可插拔认证

本节概述了如何使用 PAM 认证插件从 MySQL 客户端程序连接到服务器。以下部分提供了以特定方式使用 PAM 认证的说明。假设服务器已启用服务器端 PAM 插件，如“安装 PAM 可插拔认证”中所述。

要在 `CREATE USER` 语句的 `IDENTIFIED WITH` 子句中引用 PAM 认证插件，请使用名称 `authentication_pam`。例如：

```sql
CREATE USER user
  IDENTIFIED WITH authentication_pam
  AS 'auth_string';
```

认证字符串指定以下类型的信息：

- PAM 服务名（参见 PAM 认证 MySQL 用户的工作原理）。以下讨论中的示例使用服务名 `mysql-unix` 进行传统 Unix 密码认证，使用 `mysql-ldap` 进行 LDAP 认证。
- 对于代理支持，PAM 提供了一种方式，使 PAM 模块可以返回给服务器一个与客户端
- 程序连接时传递的外部用户名不同的 MySQL 用户名。使用认证字符串控制从外部用户名到 MySQL 用户名的映射。如果您想利用代理用户功能，认证字符串必须包含这种映射。


例如，如果一个账户使用 `mysql-unix` PAM 服务名，并且应该将操作系统用户在 `root` 和 `users` PAM 组中映射到 `developer` 和 `data_entry` MySQL 用户，使用如下语句：

```sql
CREATE USER user
  IDENTIFIED WITH authentication_pam
  AS 'mysql-unix, root=developer, users=data_entry';
```

PAM 认证插件的认证字符串语法遵循以下规则：

- 字符串由 PAM 服务名组成，后面可选地跟一个 PAM 组映射列表，该列表包含一个或多个关键字/值对，每个对指定一个 PAM 组名和一个 MySQL 用户名：
  ```pam_service_name[,pam_group_name=mysql_user_name]...```
- 插件解析使用账户的每次连接尝试的认证字符串。为了尽量减少开销，保持字符串尽可能短。
- 每个 `pam_group_name=mysql_user_name` 对必须以逗号分隔。
- 不在双引号内的前导和尾随空格会被忽略。
- 未加引号的 `pam_service_name`、`pam_group_name` 和 `mysql_user_name` 值可以包含除等号、逗号或空格以外的任何内容。
- 如果 `pam_service_name`、`pam_group_name` 或 `mysql_user_name` 值用双引号括起来，则引号之间的所有内容都是该值的一部分。这对于值包含空格字符的情况是必要的。所有字符都是合法的，除了双引号和反斜杠（`\`）。要包含任一字符，请使用反斜杠转义它。

如果插件成功认证外部用户名（客户端传递的名称），它会在认证字符串中查找 PAM 组映射列表，如果存在，则根据外部用户所属的 PAM 组，使用该列表返回不同的 MySQL 用户名：

- 如果认证字符串不包含 PAM 组映射列表，插件返回外部名称。
- 如果认证字符串包含 PAM 组映射列表，插件从左到右检查列表中的每个 `pam_group_name=mysql_user_name` 对，尝试在分配给已认证用户的组的非 MySQL 目录中找到 `pam_group_name` 值的匹配项，并返回找到的第一个匹配项的 `mysql_user_name`。如果插件未找到任何 PAM 组的匹配项，则返回外部名称。如果插件无法在目录中查找组，它会忽略 PAM 组映射列表并返回外部名称。

以下部分描述了如何设置使用 PAM 认证插件的几种认证方案：

- 无代理用户。这仅使用 PAM 检查登录名和密码。每个被允许连接到 MySQL 服务器的外部用户应具有一个与 PAM 认证的 MySQL 账户匹配的账户。（对于 `'user_name'@'host_name'` 的 MySQL 账户要匹配外部用户，`user_name` 必须是外部用户名，`host_name` 必须匹配客户端连接的主机。）认证可以通过各种 PAM 支持的方法执行。后续讨论显示如何使用传统 Unix 密码和 LDAP 密码认证客户端凭证。
- 仅代理用户，带 PAM 组映射。对于这种情况，创建一个或多个定义了不同权限集的 MySQL 账户。（理想情况下，不应直接通过这些账户连接。）然后定义一个默认用户，通过 PAM 认证，该用户使用某种映射方案（通常基于用户所属的外部 PAM 组）将所有外部用户名映射到持有权限集的几个 MySQL 账户。任何通过指定外部用户名作为客户端用户名连接的客户端都将映射到持有权限集的 MySQL 账户之一，并使用其权限。讨论显示如何使用传统 Unix 密码设置此方案，但也可以使用其他 PAM 方法，如 LDAP。

这些方案的变体是可能的：

- 您可以允许某些用户直接登录（不使用代理），但要求其他用户通过代理账户连接。
- 您可以为某些用户使用一种 PAM 认证方法，为其他用户使用另一种方法，通过在 PAM 认证的账户中使用不同的 PAM 服务名。例如，您可以对某些用户使用 `mysql-unix` PAM 服务，对其他用户使用 `mysql-ldap`。

这些示例假设以下情况。 如果您的系统设置不同，可能需要进行一些调整。

- 登录名和密码分别为 `antonio` 和 `antonio_password`。将这些更改为您要认证的用户。
- PAM 配置目录为 `/etc/pam.d`。
- PAM 服务名对应于认证方法（在本讨论中为 `mysql-unix` 或 `mysql-ldap`）。要使用给定的 PAM 服务，必须在 PAM 配置目录中设置一个同名的 PAM 文件（如果该文件不存在，则创建该文件）。此外，必须在 `CREATE USER` 语句的认证字符串中命名 PAM 服务，以便任何使用该 PAM 服务认证的账户都能认证。

PAM 认证插件在初始化时检查服务器启动环境中是否设置了 `AUTHENTICATION_PAM_LOG` 环境值。如果设置，插件启用将诊断消息记录到标准输出。根据服务器的启动方式，消息可能显示在控制台或错误日志中。这些消息对于调试插件执行认证时发生的 PAM 相关问题可能很有帮助。有关更多信息，请参见 PAM 认证调试。

##### 无代理用户的 PAM Unix 密码认证

此认证方案使用 PAM 检查以操作系统用户名和 Unix 密码定义的外部用户，无需代理。每个被允许连接到 MySQL 服务器的外部用户应具有一个与使用传统 Unix 密码存储通过 PAM 认证的 MySQL 账户匹配的账户。

> **注意**
>
> 传统 Unix 密码通过 `/etc/shadow` 文件进行检查。有关此文件的可能问题的信息，请参见“PAM 认证访问 Unix 密码存储”。

1. 验证 Unix 认证是否允许使用用户名 `antonio` 和密码 `antonio_password` 登录到操作系统。
2. 通过创建一个名为 `/etc/pam.d/mysql-unix` 的 `mysql-unix` PAM 服务文件，设置 PAM 使用传统的 Unix 密码认证 MySQL 连接。文件内容依赖于系统，因此请检查 `/etc/pam.d` 目录中现有的与登录相关的文件，了解它们的外观。在 Linux 上，`mysql-unix` 文件可能如下所示：
   ```plaintext
    #%PAM-1.0
    auth            include         password-auth
    account         include         password-auth
   ```

    对于 macOS，使用 `login` 而不是 `password-auth`。

    在某些系统上，PAM 文件格式可能不同。例如，在 Ubuntu 和其他基于 Debian 的系统上，使用以下文件内容代替：
    ```plaintext
    @include common-auth
    @include common-account
    @include common-session-noninteractive
    ```

3. 创建一个与操作系统用户名相同的 MySQL 账户，并定义其使用 PAM 插件和 `mysql-unix` PAM 服务进行认证：
   ```sql
    CREATE USER 'antonio'@'localhost'
    IDENTIFIED WITH authentication_pam
    AS 'mysql-unix';
    GRANT ALL PRIVILEGES
    ON mydb.*
    TO 'antonio'@'localhost';
   ```
   此处，认证字符串仅包含 PAM 服务名 `mysql-unix`，用于认证 Unix 密码。

4. 使用 `mysql` 命令行客户端以 `antonio` 身份连接到 MySQL 服务器。例如：
   ```bash
   $> mysql --user=antonio --password --enable-cleartext-plugin
    Enter password: antonio_password
   ```
   
   服务器应允许连接，并返回如下查询结果：

   ```sql
    mysql> SELECT USER(), CURRENT_USER(), @@proxy_user;
    +-------------------+-------------------+--------------+
    | USER()            | CURRENT_USER()    | @@proxy_user |
    +-------------------+-------------------+--------------+
    | antonio@localhost | antonio@localhost | NULL         |
    +-------------------+-------------------+--------------+
   ```

   这表明操作系统用户 `antonio` 已被认证为具有授予 `antonio` MySQL 用户的权限，且没有发生代理。

    >   **注意**
    >
    >   客户端 `mysql_clear_password` 认证插件保持密码不变，因此客户端程序将密码以明文形式发送到 MySQL 服务器。这使得密码可以按原样传递给 PAM。使用服务器端 PAM 库需要明文密码，但在某些配置中可能存在安全问题。这些措施可以将风险降至最低：
    >
    >   - 为了降低无意中使用 `mysql_clear_password` 插件的可能性，MySQL 客户端必须显式启用它（例如，通过 `--enable-cleartext-plugin` 选项）。参见 [8.4.1.4 节, “客户端明文可插拔认证”](#8.4.1.4-客户端明文可插拔认证)。
    >   - 为避免在启用 `mysql_clear_password` 插件时密码暴露，MySQL 客户端应使用加密连接连接到 MySQL 服务器。参见 [8.3.1 节, “配置 MySQL 使用加密连接”](#8.3.1-配置-mysql-使用加密连接)。
    >


  ##### 无代理用户的 PAM LDAP 认证

此认证方案使用 PAM 检查以操作系统用户名和 LDAP 密码定义的外部用户，无需代理。每个被允许连接到 MySQL 服务器的外部用户应具有一个与使用 LDAP 通过 PAM 认证的 MySQL 账户匹配的账户。

要使用 PAM LDAP 可插拔认证 MySQL，请满足以下先决条件：

  - 必须有一个可供 PAM LDAP 服务通信的 LDAP 服务器。
  - 每个要由 MySQL 认证的 LDAP 用户必须存在于由 LDAP 服务器管理的目录中。

    > **注意**
    >
    > 另一种使用 LDAP 进行 MySQL 用户认证的方法是使用特定的 LDAP 认证插件。参见 [8.4.1.7 节, “LDAP 可插拔认证”](#8.4.1.7-ldap-可插拔认证)。

按以下步骤配置 MySQL 进行 PAM LDAP 认证：

验证 Unix 认证是否允许使用用户名 `antonio` 和密码 `antonio_password` 登录到操作系统。

通过创建一个名为 `/etc/pam.d/mysql-ldap` 的 `mysql-ldap` PAM 服务文件，设置 PAM 使用 LDAP 认证 MySQL 连接。文件内容依赖于系统，因此请检查 `/etc/pam.d` 目录中现有的与登录相关的文件，了解它们的外观。在 Linux 上，`mysql-ldap` 文件可能如下所示：

  ```plaintext
  #%PAM-1.0
  auth        required    pam_ldap.so
  account     required    pam_ldap.so
  ```

  如果系统上的 PAM 对象文件有不同于 `.so` 的后缀，请替换为正确的后缀。

  在某些系统上，PAM 文件格式可能不同。

  创建一个与操作系统用户名相同的 MySQL 账户，并定义其使用 PAM 插件和 `mysql-ldap` PAM 服务进行认证：

  ```sql
  CREATE USER 'antonio'@'localhost'
    IDENTIFIED WITH authentication_pam
    AS 'mysql-ldap';
  GRANT ALL PRIVILEGES
    ON mydb.*
    TO 'antonio'@'localhost';
  ```

此处，认证字符串仅包含 PAM 服务名 `mysql-ldap`，用于通过 LDAP 认证。

连接到服务器的方式与“无代理用户的 PAM Unix 密码认证”中描述的相同。

  ##### 带代理用户和组映射的 PAM Unix 密码认证

此处描述的认证方案使用代理和 PAM 组映射，将通过 PAM 认证连接的 MySQL 用户映射到定义了不同权限集的其他 MySQL 账户。用户不直接通过定义权限的账户连接，而是通过使用 PAM 认证的默认代理账户连接，使所有外部用户都映射到持有权限集的 MySQL 账户。任何使用代理账户连接的用户都将映射到这些 MySQL 账户之一，并使用其权限。

此处显示的过程使用 Unix 密码认证。要改用 LDAP，请参见“无代理用户的 PAM LDAP 认证”的早期步骤。

  > **注意**
  >
  > 传统 Unix 密码通过 `/etc/shadow` 文件进行检查。有关此文件的可能问题的信息，请参见“PAM 认证访问 Unix 密码存储”。

验证 Unix 认证是否允许使用用户名 `antonio` 和密码 `antonio_password` 登录到操作系统。

验证 `antonio` 是否是 `root` 或 `users` PAM 组的成员。

通过创建一个名为 `/etc/pam.d/mysql-unix` 的 `mysql-unix` PAM 服务文件，设置 PAM 通过操作系统用户认证 `mysql-unix` PAM 服务。文件内容依赖于系统，因此请检查 `/etc/pam.d` 目录中现有的与登录相关的文件，了解它们的外观。在 Linux 上，`mysql-unix` 文件可能如下所示：

  ```plaintext
  #%PAM-1.0
  auth            include         password-auth
  account         include         password-auth
  ```

对于 macOS，使用 `login` 而不是 `password-auth`。

在某些系统上，PAM 文件格式可能不同。例如，在 Ubuntu 和其他基于 Debian 的系统上，使用以下文件内容代替：

  ```plaintext
  @include common-auth
  @include common-account
  @include common-session-noninteractive
  ```

创建一个默认代理用户 (`''@''`)，将外部 PAM 用户映射到代理账户：

  ```sql
  CREATE USER ''@''
    IDENTIFIED WITH authentication_pam
    AS 'mysql-unix, root=developer, users=data_entry';
  ```

此处，认证字符串包含 PAM 服务名 `mysql-unix`，用于认证 Unix 密码。认证字符串还将外部用户在 `root` 和 `users` PAM 组中映射到 `developer` 和 `data_entry` MySQL 用户名。

设置代理用户时，需要 PAM 组映射列表。否则，插件无法知道如何从外部用户名映射到适当的代理 MySQL 用户名。

  > **注意**
  >
  > 如果您的 MySQL 安装有匿名用户，它们可能与默认代理用户冲突。有关此问题的信息及其处理方法，请参见“默认代理用户和匿名用户冲突”。

创建代理账户并授予每个账户应有的权限：

  ```sql
  CREATE USER 'developer'@'localhost'
    IDENTIFIED WITH mysql_no_login;
  CREATE USER 'data_entry'@'localhost'
    IDENTIFIED WITH mysql_no_login;
  
  GRANT ALL PRIVILEGES
    ON mydevdb.*
    TO 'developer'@'localhost';
  GRANT ALL PRIVILEGES
    ON mydb.*
    TO 'data_entry'@'localhost';
  ```

代理账户使用 `mysql_no_login` 认证插件，以防止客户端直接使用这些账户登录到 MySQL 服务器。相反，使用 PAM 认证的用户应通过代理使用 `developer` 或 `data_entry` 账户，具体取决于其 PAM 组。（假设已安装插件。有关说明，请参见 [8.4.1.9 节, “无登录可插拔认证”](#8.4.1.9-无登录可插拔认证)。）有关保护代理账户免于直接使用的替代方法，请参见“防止直接登录代理账户”。

授予代理账户对每个代理账户的 PROXY 权限：

  ```sql
  GRANT PROXY
    ON 'developer'@'localhost'
    TO ''@'';
  GRANT PROXY
    ON 'data_entry'@'localhost'
    TO ''@'';
  ```

使用 `mysql` 命令行客户端以 `antonio` 身份连接到 MySQL 服务器。

  ```bash
  $> mysql --user=antonio --password --enable-cleartext-plugin
  Enter password: antonio_password
  ```

服务器使用默认 `''@''` 代理账户进行认证。`antonio` 的最终权限取决于 `antonio` 是哪个 PAM 组的成员。如果 `antonio` 是 `root` PAM 组的成员，PAM 插件将 `root` 映射到 `developer` MySQL 用户名并将该名称返回给服务器。服务器验证 `''@''` 是否对 `developer` 拥有 PROXY 权限并允许连接。如下查询返回的输出如下所示：

  ```sql
  mysql> SELECT USER(), CURRENT_USER(), @@proxy_user;
  +-------------------+---------------------+--------------+
  | USER()            | CURRENT_USER()      | @@proxy_user |
  +-------------------+---------------------+--------------+
  | antonio@localhost | developer@localhost | ''@''        |
  +-------------------+---------------------+--------------+
  ```

这表明操作系统用户 `antonio` 被认证为具有 `developer` MySQL 用户授予的权限，并且代理通过默认代理账户进行。

如果 `antonio` 不是 `root` PAM 组的成员，而是 `users` PAM 组的成员，类似的过程会发生，但插件将用户 PAM 组成员映射到 `data_entry` MySQL 用户名并将该名称返回给服务器：

  ```sql
  mysql> SELECT USER(), CURRENT_USER(), @@proxy_user;
  +-------------------+----------------------+--------------+
  | USER()            | CURRENT_USER()       | @@proxy_user |
  +-------------------+----------------------+--------------+
  | antonio@localhost | data_entry@localhost | ''@''        |
  +-------------------+----------------------+--------------+
  ```

这表明操作系统用户 `antonio` 被认证为具有 `data_entry` MySQL 用户的权限，并且代理通过默认代理账户进行。

  > **注意**
  >
  > 客户端 `mysql_clear_password` 认证插件保持密码不变，因此客户端程序将密码以明文形式发送到 MySQL 服务器。这使得密码可以按原样传递给 PAM。使用服务器端 PAM 库需要明文密码，但在某些配置中可能存在安全问题。这些措施可以将风险降至最低：
  >
  > - 为了降低无意中使用 `mysql_clear_password` 插件的可能性，MySQL 客户端必须显式启用它（例如，通过 `--enable-cleartext-plugin` 选项）。参见 [8.4.1.4 节, “客户端明文可插拔认证”](#8.4.1.4-客户端明文可插拔认证)。
  > - 为避免在启用 `mysql_clear_password` 插件时密码暴露，MySQL 客户端应使用加密连接连接到 MySQL 服务器。参见 [8.3.1 节, “配置 MySQL 使用加密连接”](#8.3.1-配置-mysql-使用加密连接)。

##### PAM 认证访问 Unix 密码存储

在某些系统上，Unix 认证使用密码存储，如 `/etc/shadow`，这是一个通常具有受限访问权限的文件。这可能导致 MySQL 基于 PAM 的认证失败。不幸的是，PAM 实现无法区分“无法检查密码”（例如，由于无法读取 `/etc/shadow`）和“密码不匹配”。如果您使用 Unix 密码存储进行 PAM 认证，您可以通过以下方法之一来启用 MySQL 对其的访问：

- 假设 MySQL 服务器是从 `mysql` 操作系统账户运行的，将该账户添加到具有 `/etc/shadow` 访问权限的 `shadow` 组：

  1. 在 `/etc/group` 中创建 `shadow` 组。
  2. 将 `mysql` 操作系统用户添加到 `/etc/group` 中的 `shadow` 组。
  3. 将 `/etc/group` 分配给 `shadow` 组并启用组读权限：
    ```bash
    chgrp shadow /etc/shadow
    chmod g+r /etc/shadow
    ```
  
   4. 重新启动 MySQL 服务器。

- 
  如果您使用 `pam_unix` 模块和 `unix_chkpwd` 实用程序，按以下步骤启用密码存储访问：

  ```bash
  chmod u-s /usr/sbin/unix_chkpwd
  setcap cap_dac_read_search+ep /usr/sbin/unix_chkpwd
  ```

   根据您的平台调整 `unix_chkpwd` 的路径。

##### PAM 认证调试

PAM 认证插件在初始化时检查 `AUTHENTICATION_PAM_LOG` 环境值是否设置。在 MySQL 8.0.35 及更早版本中，值不重要。如果设置，插件启用将诊断消息记录到标准输出。这些消息可能对调试插件执行认证时发生的 PAM 相关问题很有帮助。请注意，在这些版本中，消息中包含密码。

从 MySQL 8.0.36 开始，设置 `AUTHENTICATION_PAM_LOG=1`（或其他任意值）会生成相同的诊断消息，但不包含任何密码。如果您希望在这些消息中包含密码，请设置 `AUTHENTICATION_PAM_LOG=PAM_LOG_WITH_SECRET_INFO`。

一些消息包括对 PAM 插件源文件和行号的引用，这使得插件动作与它们发生的代码位置更加紧密地联系起来。

调试连接失败并确定连接尝试期间发生情况的另一种技术是将 PAM 认证配置为允许所有连接，然后检查系统日志文件。此技术应仅在临时基础上使用，不应在生产服务器上使用。

配置一个名为 `/etc/pam.d/mysql-any-password` 的 PAM 服务文件，内容如下（在某些系统上格式可能不同）：

```plaintext
#%PAM-1.0
auth        required    pam_permit.so
account     required    pam_permit.so
```

创建一个使用 PAM 插件并命名 `mysql-any-password` PAM 服务的账户：

```sql
CREATE USER 'testuser'@'localhost'
  IDENTIFIED WITH authentication_pam
  AS 'mysql-any-password';
```

`mysql-any-password` 服务文件导致任何认证尝试返回 true，即使密码不正确。如果认证尝试失败，这表示问题出在 MySQL 方面。否则，问题出在操作系统/PAM 方面。要查看可能发生的情况，请检查系统日志文件，如 `/var/log/secure`、`/var/log/audit.log`、`/var/log/syslog` 或 `/var/log/messages`。

确定问题后，删除 `mysql-any-password` PAM 服务文件以禁用任何密码访问。
